## Section 1: Modules (Questions 1-12)

### Q1
You added a `module "vpc"` block sourced from the public registry. After running `terraform plan`, you get an error saying "Module not installed". What should you do?

A. Run `terraform validate`
B. Run `terraform get` only
C. Run `terraform init`
D. Manually copy the module to `.terraform/modules/`

<details>
<summary>Answer</summary>

**C. Run `terraform init`**

`terraform init` downloads modules from remote sources into `.terraform/modules/`. While `terraform get` also downloads modules, `terraform init` is the standard command and handles providers + modules + backend together.
</details>

### Q2
A child module `./modules/db` has this code. From your root module, how do you access the database endpoint?

```hcl
# Inside ./modules/db/main.tf
resource "aws_db_instance" "main" {
  # ...
}
```

A. `aws_db_instance.main.endpoint`
B. `module.db.aws_db_instance.main.endpoint`
C. `module.db.endpoint` — but only if an output is declared
D. `var.db.endpoint`

<details>
<summary>Answer</summary>

**C. `module.db.endpoint` — but only if an output is declared**

The child module MUST declare an `output` block. Without it, the parent has zero visibility into child resources. Direct access to child resources (option B) is never allowed.
</details>

### Q3
You're calling a registry module without a `version` argument:

```hcl
module "vpc" {
  source = "terraform-aws-modules/vpc/aws"
}
```

What is the most accurate description of what happens?

A. Terraform errors and refuses to run
B. Terraform uses the latest available version, which may cause unexpected upgrades
C. Terraform pins to version 1.0.0 by default
D. Terraform uses the version from the lock file only

<details>
<summary>Answer</summary>

**B. Terraform uses the latest available version, which may cause unexpected upgrades**

The `version` argument is **optional but strongly recommended**. Without it, Terraform downloads the latest version each time, which can introduce breaking changes.
</details>

### Q4
You have two child modules `network` and `app`. The `app` module needs the VPC ID created in `network`. How do you wire this correctly?

A. In `app` module, reference `module.network.vpc_id` directly
B. In root, declare network's output as an input to app
C. Use a `data` source in `app` to query the VPC
D. Add `depends_on = [module.network]` in app

<details>
<summary>Answer</summary>

**B. In root, declare network's output as an input to app**

Child modules cannot reference each other directly. Root acts as the bridge:
```hcl
module "app" {
  vpc_id = module.network.vpc_id
```

</details>

### Q7
A module includes this version constraint: `version = "~> 3.5"`. Which versions are acceptable?

A. 3.5.0 only
B. 3.5.0 through 3.9.x but NOT 4.0.0
C. 3.5.0 through 4.x
D. Any version greater than 3.5.0

<details>
<summary>Answer</summary>

**B. 3.5.0 through 3.9.x but NOT 4.0.0**

`~> 3.5` means `>= 3.5, < 4.0`. The pessimistic operator allows minor and patch updates but NOT major version bumps.

Note: `~> 3.5.0` would be more restrictive: `>= 3.5.0, < 3.6.0`.
</details>

### Q8
You want to use the same module twice in your config to create resources in two AWS regions. What must you do?

A. Define the module twice with different `alias` attributes
B. Use `count = 2` on the module block
C. Use two `module` blocks with different names and pass different `providers` to each
D. Modules cannot create resources in multiple regions

<details>
<summary>Answer</summary>

**C. Use two `module` blocks with different names and pass different `providers` to each**

```hcl
provider "aws" { alias = "us_east"; region = "us-east-1" }
provider "aws" { alias = "us_west"; region = "us-west-2" }

module "vpc_east" {
  source    = "./modules/vpc"
  providers = { aws = aws.us_east }
}

module "vpc_west" {
  source    = "./modules/vpc"
  providers = { aws = aws.us_west }
}
```
</details>

### Q12
A new junior engineer puts this in a child module's `variables.tf`. What's the issue?

```hcl
variable "instance_count" {

}

variable "environment" {
  # missing type
```

## Section 3: State Management (Questions 23-32)

### Q23
You accidentally deleted `terraform.tfstate` from your local backend. What's the most serious immediate consequence?

A. Terraform will recreate the state file from configuration
B. Terraform loses track of managed resources and may try to recreate them
C. Terraform will fetch state from the cloud provider
D. The infrastructure is automatically destroyed

<details>
<summary>Answer</summary>

**B. Terraform loses track of managed resources and may try to recreate them**

State is the mapping between config and real resources. Without it, Terraform thinks nothing exists and tries to create duplicates on next apply.
</details>

### Q24
A resource was modified manually in the AWS console. You want to update Terraform state to match reality WITHOUT making changes. Which command?

D. `terraform state pull`

<details>
<summary>Answer</summary>

**B. `terraform apply -refresh-only`**

`terraform plan -refresh-only` shows the drift but doesn't update state. `apply -refresh-only` actually writes the changes to state without modifying real infrastructure.

### Q43
What is the correct order of variable precedence (lowest to highest)?

A. Defaults → tfvars → CLI flags → env vars
B. Defaults → tfvars → auto.tfvars → -var-file → -var → env vars
C. env vars → CLI → tfvars → defaults

<details>
<summary>Answer</summary>

**D. Defaults → env vars → tfvars → auto.tfvars → -var-file → -var**

Wait — the official docs put environment variables at the bottom. Let me correct:

1. Variable defaults
2. Environment variables (`TF_VAR_*`)
3. `terraform.tfvars`

So `-var` flag wins over everything.
</details>

You're decommissioning ONE Cloud SQL database from a config that has 50 resources. What's the best approach?

A. `terraform destroy -target=google_sql_database_instance.main`
B. Delete the resource block from config and run `terraform apply`
C. Use a `removed` block in the config
D. Both B and C are valid; targeting is discouraged

<details>
<summary>Answer</summary>

**D. Both B and C are valid; targeting is discouraged**

Either delete the block (declarative) or use a `removed` block (Terraform 1.7+). `-target` is discouraged for routine work because it bypasses the dependency graph.
</details>

### Q49
Why does `terraform fmt` exist?

A. To validate config syntax
B. To rewrite config files in canonical formatting/style
C. To format the state file
D. To compress files

<details>
<summary>Answer</summary>

**B. To rewrite config files in canonical formatting/style**

`terraform fmt` ensures consistent indentation, alignment, and style. Run it before commits. Returns 0 on success.

### Q50
True or False: Rearranging the order of resource blocks within `.tf` files affects the order Terraform creates them.

<details>
<summary>Answer</summary>

**False.**

Terraform builds a dependency graph based on resource references and `depends_on`. The textual order of blocks in files is irrelevant. This is a key property of declarative IaC.
</details>
